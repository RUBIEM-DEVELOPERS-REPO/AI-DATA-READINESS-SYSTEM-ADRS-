--------------------------------------------------
B. KNOWLEDGE-GRAPH-READY DATASET
--------------------------------------------------

GOAL
Produce ONE coherent graph artifact that is ready for analysis by existing graph systems.

IMPORTANT
The final graph output must NOT be fragmented in a way that makes graph analysis difficult.
It should be analysis-ready for tools such as:
- Neo4j
- Memgraph
- Neptune
- Graphistry
- TigerGraph
- generic graph loaders

FINAL OUTPUT FORMAT
Export one unified graph dataset:
kg_graph.jsonl

Each record must be either:
- NODE
or
- EDGE

Example structure:

{
  "record_type": "NODE",
  "id": "PARTY-001",
  "label": "PARTY",
  "properties": {...},
  "provenance": {...}
}

{
  "record_type": "EDGE",
  "id": "EDGE-001",
  "type_label": "CUSTOMER_OF",
  "from": "PARTY-123",
  "to": "PARTY-456",
  "properties": {...},
  "provenance": {...}
}

GRAPH REQUIREMENTS
1. Canonical node labels only:
- PARTY
- DOCUMENT
- TRANSACTION or EVENT
- OBJECT (only if truly needed)
- PRODUCT / ASSET if supported

2. Eliminate fragmented/duplicate entities.
Use deterministic rules plus the existing model to improve entity consolidation:
- normalize names
- remove titles/noise
- reconcile merchant variants
- detect likely same customer / same organization
- produce merge candidates or merged canonical nodes

3. Do not emit weak fragment nodes like:
- single tokens
- accidental OCR fragments
- partial role words
unless clearly justified

4. Identifier handling
- normalize all identifiers
- canonicalize identifier types
- do not concatenate multi-values
- attach identifiers properly to relevant node types
- use identifiers to support merge logic

5. Relationship correctness
- prevent invalid self-loops
- enforce type-compatible edges
- replace weak generic edges with meaningful semantic relations
- do not publish low-confidence nonsense edges as normal graph facts

6. Provenance is mandatory
Every node and edge should carry as much of the following as possible:
- evidence_id
- document_id
- evidence_pointer_ids
- confidence
- provenance_quality

7. Use the existing model to improve graph quality in specific tasks:
- role inference (merchant vs customer vs issuer)
- organization/person disambiguation
- fuzzy entity consolidation support
- semantic edge labeling when rule-based signals are ambiguous

8. Final graph must be one coherent analysis-ready dataset, not just raw intermediary exports.

RAG REQUIREMENTS
1. Replace mechanical page slicing with semantic chunking.
Use the existing system model to detect semantic chunk boundaries.
Chunk by meaningful sections such as:
- issuer header
- customer details
- payment details
- line items
- totals
- instructions
- terms

2. Remove trivial chunks:
- label-only fragments
- broken OCR scraps
- isolated meaningless numbers
unless contextually important

3. Ground every chunk
- populate evidence_pointer_ids
- link chunk to source evidence precisely
- use page_number + pointer ids + spans

4. Replace placeholder AUTO-* entity links
Use canonical entity ids where available.
If unresolved, clearly mark unresolved candidates instead of pretending they are final.

5. Make validation state meaningful
Support:
- VALIDATED
- PARTIALLY_VALIDATED
- UNVALIDATED

6. Detect and manage repeated boilerplate
- repeated headers
- repeated payment instructions
- repeated signatures
Mark template-like content so downstream retrieval systems can down-rank it.

7. Governance-aware output
Support:
- INTERNAL_RAW
- INTERNAL_STRUCTURED
- REDACTED
Use the existing model plus rules to detect:
- names
- phones
- emails
- bank details
- account numbers
Populate contains_pii and redaction_status accordingly.
